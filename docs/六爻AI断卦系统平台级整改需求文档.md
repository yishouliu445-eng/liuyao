# 六爻AI断卦系统平台级整改需求文档

> **文档版本**：1.0  
> **最后更新**：2026-04-13  
> **状态**：Proposed  
> **基线版本**：当前仓库 `v2.0`（Session、多轮分析、应验反馈、Prompt 回归、Replay 能力已落地）  
> **关联文档**：[产品需求文档](./六爻AI断卦系统产品需求文档.md) · [技术设计文档](./六爻AI断卦系统设计文档.md) · [基础设施与质量保障设计文档](./六爻AI断卦系统基础设施与质量保障设计文档.md)

## 一、文档目的

本文件用于定义“平台级重构”阶段的整改需求，目标不是重复描述已经完成的 `v2.0` 能力，而是基于当前真实代码基线，明确仍然存在的结构性问题、整改边界、验收标准与阶段目标。

本次整改的核心判断是：

1. 当前系统已经从“单次排盘工具”升级到“Session 多轮分析 + 应验反馈”的 `v2.0` 形态。
2. 现有实现已具备可用的产品闭环，但平台能力仍存在若干“局部可用、整体不稳”的断点。
3. 若只做补丁式修复，会继续放大分析链路分叉、证据不可追溯、降级不可观测、运行时能力单机化等问题。
4. 因此需要一次“平台级重构”，统一分析契约、证据契约、质量门禁与运行时治理能力。

---

## 二、当前基线与问题定义

### 2.1 当前已具备的能力

当前仓库已经落地的能力包括：

- `Session` 多轮会话能力，支持创建、追问、查询、关闭。
- `OrchestratedAnalysisService` 结构化 JSON 输出。
- `PromptRegressionTest` 黄金数据集回归。
- `Case replay` 差异比对与 replay run 持久化。
- `VerificationEventService` 应验事件生成、反馈、提醒、过期处理。
- `liuyao-worker` 知识分块、向量化、全文检索基础能力。
- Docker Compose、CI、前端构建链路。

这意味着整改不能以“从零设计新系统”为前提，而必须以“保护已上线主能力”为前提进行。

### 2.2 仍然成立的关键问题

#### 问题 A：分析链路存在双轨制，平台契约不统一

当前系统存在两套并存的分析表达路径：

- 新主链路：`SessionService -> OrchestratedAnalysisService -> AnalysisOutputDTO`
- 兼容链路：`DivinationService -> AnalysisService -> 机械文本/润色文本`

问题在于：

- 同一问题可能得到两种不同粒度、不同约束级别的输出。
- 旧接口的兼容逻辑仍然回落到旧 `AnalysisService` 文本链路。
- 前端与外部调用方难以依赖一个稳定的“统一分析契约”。

#### 问题 B：RAG 证据仍停留在“字符串片段”层，而不是一等对象

当前 `knowledgeSnippets` 主要以字符串形式向上游传递。虽然可用于 Prompt 组装，但仍缺少：

- `bookId / chunkId / source / chapter / knowledgeType` 等稳定来源字段
- 每条结论与证据之间的结构化引用关系
- 检索结果、最终引用结果、落库结果之间的一致性约束

其直接后果是：

- 前端无法做真正可点击、可回溯的证据链展示
- LLM 虽然“被要求引用”，但服务端无法严格验证是否真的引用了检索结果
- replay 与质量评估无法精确回答“哪些引用有效、哪些引用是幻觉”

#### 问题 C：LLM 当前有结构校验，但缺少语义校验与失败分型

当前系统已经具备 JSON 结构输出与解析失败回退能力，但还缺少：

- 规则结论与 LLM 结论是否一致的语义校验
- `useGod`、`effectiveResultLevel`、引用来源等关键字段的一致性校验
- 降级原因的结构化暴露，而不是仅写日志
- 不同失败类型的分型：超时、空响应、JSON 解析失败、语义冲突、证据冲突、安全拒答

这会导致：

- 调用方只知道“有结果”，不知道结果是完整分析还是降级结果
- 系统难以量化“平台到底是哪里在退化”
- 后续评估体系缺少真实运行数据

#### 问题 D：质量体系碎片化，无法形成统一平台门禁

当前质量能力分散在多个点上：

- Prompt regression 已有
- case replay 已有
- Session API 集成测试已有
- worker 测试已有

但仍缺少统一的质量平台，将以下对象纳入同一个评估框架：

- Prompt 版本
- 规则版本
- 用神配置版本
- RAG 召回版本
- LLM 模型版本
- 最终输出质量评分

因此系统难以回答：

- 一次改动到底提升了 Prompt、规则，还是只是换了模型偶然更好
- RAG 召回差是检索问题、chunk 问题还是 Prompt 问题
- 应验反馈是否能够反哺到规则和 Prompt 的治理

#### 问题 E：运行时治理仍然偏单机实现，缺少平台级能力

当前存在若干“可用但不可扩”的实现：

- 限流基于进程内内存计数，无法跨实例共享
- 定时任务基于单服务本地 `@Scheduled`
- 调用链缺少统一 trace id、执行快照与降级事件台账
- 平台配置版本缺少统一登记与审计

这会在未来扩容、灰度、回放、排障时迅速成为瓶颈。

#### 问题 F：高敏感占测场景的安全策略尚未平台化

系统已在 Prompt 中做了宿命论约束，但对健康、财务、官司等场景仍缺少更强的服务端治理能力：

- 风险问题识别
- 输出模板级别的免责声明与限制
- 超范围问题的保守回答策略
- 安全拒答与安全降级的统一记录

## 三、为什么这次要做“平台级重构”

平台级重构不是为了把当前系统“拆成更多服务”，而是为了把当前松散分布在各个模块里的能力，上升为一组统一、稳定、可验证的平台契约。

本次重构的本质目标是：

- 把“分析”从一个服务方法，升级为一个可追踪的执行平台
- 把“证据”从一个字符串数组，升级为一个可验证的引用平台
- 把“回归测试”从几个分散测试类，升级为一个统一质量平台
- 把“限流/定时/降级”从单点实现，升级为一个平台运行时能力

## 四、目标与非目标

### 4.1 目标

#### G1：建立单一分析平台契约

无论调用来源是：

- `POST /api/sessions`
- `POST /api/sessions/{id}/messages`
- `POST /api/divinations/analyze`
- replay / regression / evaluation

都应收敛到同一套分析执行契约，至少共享：

- 请求结构
- 确定性事实快照
- RAG 证据结果
- LLM 输出结果
- 验证结果
- 降级记录
- 最终展示层结果

#### G2：建立证据一等公民能力

系统输出中的古籍引用必须可追溯到知识库实体，而不是停留在字符串层面。

#### G3：建立语义可验证的 AI 输出治理能力

平台必须能够区分：

- “格式正确但语义错误”
- “结构正确但引用伪造”
- “结论方向偏离规则”
- “安全策略不满足”

并对这些错误做分类与回退。

#### G4：建立统一质量评估平台

让 Prompt、规则、RAG、模型、反馈数据进入同一个评估坐标系。

#### G5：建立可扩容的运行时能力

将限流、任务调度、降级事件、配置版本和执行审计从单机实现升级为平台能力。

#### G6：保守迁移，不破坏当前可用能力

平台级重构必须分阶段推进，优先通过“模块化单体 + 清晰契约”实现，而不是在第一阶段就强拆成物理微服务。

### 4.2 非目标

本阶段明确不做以下事项：

- 不用端到端 LLM 推翻既有规则引擎
- 不在第一阶段引入复杂服务网格或消息中间件依赖
- 不在第一阶段做多租户计费平台
- 不把 worker、app、frontend 全部打散成独立团队级平台
- 不为“看起来平台化”而过度设计

## 五、整改范围

本次整改按 5 个平台能力域展开。

### 5.1 分析执行平台

#### 需求目标

将“排盘 -> 规则 -> 检索 -> LLM -> 校验 -> 输出 -> 持久化”统一为单次执行模型。

#### 功能需求

- `FR-AN-001`：平台必须定义统一的 `AnalysisExecutionRequest` 与 `AnalysisExecutionResult`。
- `FR-AN-002`：平台必须区分“确定性事实层”“证据层”“LLM 解释层”“展示层”。
- `FR-AN-003`：旧 `POST /api/divinations/analyze` 必须通过适配器接入统一执行平台。
- `FR-AN-004`：平台必须为每次分析生成稳定的 `executionId`。
- `FR-AN-005`：平台必须记录每一步执行的成功、失败、耗时、输入版本与输出摘要。
- `FR-AN-006`：平台必须支持首次分析、追问分析、replay 分析三种模式，但共享统一契约。
- `FR-AN-007`：平台必须保留“展示文本兼容输出”，用于兼容旧前端和旧接口。

### 5.2 证据与知识平台

#### 需求目标

让 RAG 结果从“面向 Prompt 的字符串片段”升级为“面向平台治理的结构化证据对象”。

#### 功能需求

- `FR-EV-001`：检索结果必须返回 `chunkId`、`bookId`、`sourceTitle`、`chapterTitle`、`knowledgeType`、`content`、`score`、`rank`。
- `FR-EV-002`：最终分析结果中的每条 `classicReference` 必须关联到至少一个真实检索命中对象。
- `FR-EV-003`：当无有效证据时，平台必须明确标记“无证据引用”，而不是交给模型自由发挥。
- `FR-EV-004`：追问检索必须允许带入“当前追问 + 原始问题 + 问类 + 用神 + 规则摘要”的复合查询。
- `FR-EV-005`：平台必须支持将“检索命中”和“最终被引用”区分开来。
- `FR-EV-006`：平台必须把证据对象落库，支持前端点击查看原文。
- `FR-EV-007`：平台必须能对 RAG 召回质量做离线评估。

### 5.3 输出验证与安全治理平台

#### 需求目标

在已有 JSON 结构校验基础上，补齐语义验证、安全校验和失败分型。

#### 功能需求

- `FR-GV-001`：平台必须校验 LLM 输出与规则结论方向是否一致。
- `FR-GV-002`：平台必须校验 `useGod`、问类、关键卦象字段是否被错误篡改。
- `FR-GV-003`：平台必须校验引用来源是否来自本次真实检索集合。
- `FR-GV-004`：平台必须校验高风险问题是否带有规定的安全提示。
- `FR-GV-005`：平台必须对失败进行分类：`TIMEOUT`、`EMPTY_RESPONSE`、`JSON_INVALID`、`SEMANTIC_MISMATCH`、`CITATION_MISMATCH`、`SAFETY_REJECTED`、`RAG_UNAVAILABLE` 等。
- `FR-GV-006`：平台必须将失败与回退信息返回给上游调用方的结构化元数据，而不是只写日志。
- `FR-GV-007`：平台必须支持不同级别的降级结果：`FULL`、`PARTIAL_WITHOUT_RAG`、`PARTIAL_WITH_FALLBACK_TEXT`、`SAFE_REFUSAL`。

### 5.4 质量与评估平台

#### 需求目标

把 prompt regression、case replay、RAG 评估、生产反馈纳入统一质量平台。

#### 功能需求

- `FR-QA-001`：平台必须支持以“数据集 + 平台版本快照 + 评估规则”的方式执行离线评估。
- `FR-QA-002`：平台必须统一记录以下版本：`promptVersion`、`ruleBundleVersion`、`ruleDefinitionsVersion`、`useGodRulesVersion`、`retrievalVersion`、`modelVersion`。
- `FR-QA-003`：平台必须支持 replay 对比，并输出差异原因。
- `FR-QA-004`：平台必须支持 Prompt regression 与 replay 进入同一评估报告。
- `FR-QA-005`：平台必须新增 RAG 召回质量评估，至少覆盖命中率、有效引用率、无证据幻觉率。
- `FR-QA-006`：平台必须为生产应验反馈提供回灌入口，作为后续评估样本。
- `FR-QA-007`：CI 必须能基于改动范围选择性触发相关评估。

### 5.5 运行时与运维平台

#### 需求目标

将当前单机实现的限流、调度、配置与可观测性能力升级为平台级治理能力。

#### 功能需求

- `FR-OP-001`：限流必须迁移到可持久化或可共享存储方案，支持多实例一致行为。
- `FR-OP-002`：计划任务必须具备防重入与租约能力，避免多实例重复执行。
- `FR-OP-003`：每次分析必须带有 trace id / execution id。
- `FR-OP-004`：平台必须记录关键指标：总耗时、检索耗时、LLM 耗时、降级率、引用率、命中率。
- `FR-OP-005`：平台必须具备配置版本登记能力，便于回放与审计。
- `FR-OP-006`：平台必须支持后续灰度切换分析策略。
- `FR-OP-007`：平台必须保留对外部模型调用成本的统计能力。

## 六、非功能需求

### 6.1 正确性

- `NFR-001`：平台返回的结论方向不得违背规则层的确定性结果。
- `NFR-002`：当证据不存在时，平台不得伪造古籍引用。
- `NFR-003`：任何降级都必须可被上游识别。

### 6.2 性能

- `NFR-004`：首次分析接口 P95 小于 12 秒。
- `NFR-005`：追问接口 P95 小于 8 秒。
- `NFR-006`：在 RAG 不可用场景下，平台必须在 3 秒内给出降级结果。

### 6.3 可观测性

- `NFR-007`：所有分析执行必须可通过 `executionId` 全链路追踪。
- `NFR-008`：错误、降级、拒答必须有可检索台账。

### 6.4 可维护性

- `NFR-009`：重构后必须降低旧分析链路与新链路的分叉点。
- `NFR-010`：模块边界必须支持后续按能力域拆分服务。

### 6.5 安全与合规

- `NFR-011`：高风险问类必须输出统一安全策略结果。
- `NFR-012`：平台不得在日志中暴露密钥或敏感原文。

### 6.6 成本控制

- `NFR-013`：平台必须统计不同模型、不同链路、不同问类的调用成本。
- `NFR-014`：平台应支持按策略降级到更轻量模型或无 RAG 模式。

## 七、业务验收标准

### 7.1 P0 验收

- `AC-P0-001`：Session 与 legacy analyze 均返回统一分析信封中的兼容子集。
- `AC-P0-002`：每次分析均生成 `executionId`。
- `AC-P0-003`：每次分析均可输出结构化 `degradation` 信息。
- `AC-P0-004`：引用内容可定位到真实 `book_chunk`。
- `AC-P0-005`：对健康、官司、财务类问题，输出含统一安全策略结果。

### 7.2 P1 验收

- `AC-P1-001`：Prompt regression、case replay、RAG 评估进入统一评估报告。
- `AC-P1-002`：限流与定时任务具备多实例一致性。
- `AC-P1-003`：前端可点击查看引用原文来源。
- `AC-P1-004`：平台可回答“本次结果是否降级、为何降级、引用了哪些证据、对应版本是什么”。

### 7.3 P2 验收

- `AC-P2-001`：基于生产反馈形成周期性回放评估。
- `AC-P2-002`：支持按平台版本进行灰度和回滚。
- `AC-P2-003`：具备按能力域拆分独立服务的准备度，但不强制当期拆服务。

## 八、分阶段交付路线

### Phase 1：平台契约统一

目标：先把分析、证据、降级的契约统一起来。

交付物：

- 统一分析执行信封
- legacy endpoint 适配器
- 结构化 degradation 返回
- 结构化 evidence 引用对象
- 基础语义校验器

### Phase 2：质量平台与运行时平台

目标：让平台变得可验证、可扩容、可排障。

交付物：

- 统一评估运行模型
- RAG 评估与 replay 评估汇总
- 持久化限流
- 租约化调度
- execution trace / metrics / audit

### Phase 3：平台抽象稳定与后续拆分准备

目标：让平台具备进一步拆分服务的清晰边界。

交付物：

- 能力域清晰的包结构与接口
- 平台配置版本管理
- 灰度策略与回滚策略
- 面向外部 API / 内部服务的双契约准备

---

## 九、风险与约束

### 9.1 风险

- 平台化过程中容易过度设计，偏离当前仓库节奏。
- 若直接物理拆服务，调试和迁移成本会显著放大。
- 若不先统一 legacy 与 session 契约，重构会一直被兼容逻辑拖住。
- 若不把 evidence 变成结构化对象，后续所有“可信引用”“RAG 评估”都会停留在表面。

### 9.2 约束

- 必须保留当前 `Session` 能力可用。
- 必须保留 `POST /api/divinations/analyze` 的兼容接口。
- 必须兼容现有 `book_chunk`、`case replay`、`PromptRegressionTest` 资产。
- 必须优先采用“模块化单体 + 可提取边界”，而非一开始就物理拆分。

---

## 十、结论

这次整改不是“再做一次功能升级”，而是把当前已经初具规模的 `v2.0` 系统，提升为真正具备平台能力的分析基础设施。

平台级重构的最小成功标准，不是“模块数量更多”，而是做到以下四件事：

1. 所有分析入口共享同一执行契约。
2. 所有古籍引用都能追溯到真实证据对象。
3. 所有降级与失败都可被结构化识别。
4. 所有规则、Prompt、RAG、模型变更都能进入统一评估闭环。

若以上四点达成，系统才能从“功能可用”进入“平台可信、可演进、可扩展”的下一阶段。
