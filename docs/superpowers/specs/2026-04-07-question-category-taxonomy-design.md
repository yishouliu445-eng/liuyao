# 六爻问类体系设计

## 目标

把当前分散在 `QuestionIntentResolver`、知识召回、分析模板中的问类字符串收敛成一套可维护的“问类体系 + 规范化入口”，避免继续靠零散 `switch/case` 扩展。

## 当前问题

当前工程里已经打通的问类只有：

- 收入
- 感情
- 工作
- 健康
- 出行
- 合作

但规则引擎和用神选择内部其实已经有更多语义：

- 求职机会
- 工作稳定
- 压力
- 人际关系
- 成长
- 考试

现状导致：

1. API 传入的 `questionCategory` 缺少统一口径
2. 同义词会散落到 `QuestionIntentResolver`
3. 分析模板和知识召回只能写死少数中文分类
4. 后续扩展“求职 / 升职 / 婚姻 / 财运 / 学业 / 官司”时容易到处改

## 设计原则

1. 保持最小 diff，不重构主链路
2. 先做“规范化 + 别名映射”，不一次性引入复杂配置中心
3. 对外接口先收敛为稳定中文问类
4. 对内仍允许继续复用现有 `QuestionIntent`
5. 新体系优先服务：
   - 用神选择
   - 分析模板
   - 知识召回

- 先不在这一轮把“家宅 / 诉讼 / 寻物”接入用神规则，只在文档中保留位置

## 实现方案

### 1. 新增问类规范化器

新增一个轻量组件，例如 `QuestionCategoryNormalizer`，职责只有两个：

1. 把用户输入的中文别名规范化成标准问类
2. 给分析和知识召回提供稳定的一级大类

示例：

- `工资 / 财运 / 薪资 / 奖金` -> `收入`
- `面试 / offer / 找工作 / 入职` -> `求职`
- `恋爱 / 婚姻 / 复合 / 对象` -> `感情`
- `旅行 / 差旅 / 航班 / 行程` -> `出行`

### 2. 保持 QuestionIntent 不大改

`QuestionIntentResolver` 继续输出现有枚举，但先走问类规范化，再映射到 intent。

这样可以避免：

- 一次性修改 `use_god_rules.json`
- 大面积重写测试

### 3. 优先改三条消费链

- `DivinationMapper`
  - 进入领域层前先规范化 `questionCategory`
- `KnowledgeSearchService`
  - 用规范化后的问类决定召回偏好
- `AnalysisService`
  - 用规范化后的问类决定结论提示语和段落语气

## 非目标

这一轮不做：

- 用数据库维护整套问类字典
- 用 LLM 自动识别问类
- 一次性补齐所有六爻场景
- 大规模改动 `QuestionIntent` 或 `use_god_rules.json`

## 验收标准

1. 新增问类规范化文档
2. `求职 / 工作 / 收入 / 压力 / 人际 / 成长 / 考试 / 感情 / 健康 / 出行 / 合作` 有统一标准名
3. `DivinationMapper` 进入领域层前会规范化 `questionCategory`
4. `QuestionIntentResolver` 能识别更多同义分类，但对现有测试兼容
5. 分析和知识召回不再只针对 6 个旧类别写死
6. 增加对应回归测试
