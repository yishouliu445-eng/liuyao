# 六爻系统开发避坑与工程原则

> 适用于：六爻辅助研判系统（Java + Python + AI 架构）
> 目标：避免关键架构错误，保证系统可扩展、可验证、可复盘

---

# 一、核心原则总览

1. 不手写底层历法（避免基础错误）
2. 排盘结果必须可验证
3. ChartSnapshot 是唯一真相
4. 规则引擎“先死后活”
5. AI 必须受约束，不能自由发挥
6. 所有分析必须可追溯
7. 系统必须可降级

---

# 二、排盘引擎原则

## 2.1 不手写万年历逻辑

禁止：
- 自己实现干支历
- 自己写节气换算
- 手写农历转换

建议：
- 使用成熟库（如 hutool 等）
- 或稳定的历法算法库

---

## 2.2 必须建立标准测试集

做一份对照数据：

- 20~50 个固定时间样例
- 对比 2~3 个排盘网站
- 固定输入 → 固定输出

验证字段：

- 日辰
- 月建
- 空亡
- 本卦
- 变卦
- 世应
- 六亲
- 六神

---

## 2.3 排盘必须可回放

同样输入 → 永远输出一致结果

---

# 三、ChartSnapshot 原则

## 3.1 唯一数据契约

所有模块依赖：

- 排盘引擎输出
- 规则引擎输入
- AI 分析输入
- 卦例存储

---

## 3.2 必须包含字段

- 本卦 / 变卦
- 每一爻：
  - 阴阳
  - 动爻
  - 六亲
  - 六神
  - 干支
- 世应
- 日辰
- 月建
- 空亡

---

## 3.3 预留扩展字段

```json
{
  "ext": {}
}
```

用于未来扩展：
- 神煞
- 暗动
- 伏神
- 调试数据

---

## 3.4 必须记录版本

```json
{
  "snapshotVersion": "v1",
  "calendarVersion": "v1"
}
```

---

# 四、规则引擎原则

## 4.1 规则必须代码化

不要：
- DSL
- 配置驱动
- AI 判规则

第一阶段：
- 每条规则一个类

---

## 4.2 输出结构化 RuleHit

禁止只返回 true/false

必须包含：

```json
{
  "ruleCode": "",
  "hit": true,
  "targets": [],
  "evidence": {}
}
```

---

## 4.3 规则必须可单测

每条规则：
- 输入固定 ChartSnapshot
- 输出固定 RuleHit

---

## 4.4 Java 不做解释

Java 只负责：
- 命中
- 证据

AI 才负责解释。

---

# 五、AI 与 RAG 原则

## 5.1 AI 不负责“算”

AI 只负责：
- 解释
- 总结
- 组织语言

---

## 5.2 Prompt 必须包含 4 类信息

1. 用户问题
2. 排盘结构
3. 规则命中
4. 知识引用

---

## 5.3 强制引用来源

AI 输出必须标明：
- 引用哪本书
- 哪段内容

---

## 5.4 防止胡说八道

禁止：
- 只给“卦名”
- 只给“问题”

---

## 5.5 AI 必须可降级

模型挂了时：

系统仍应返回：
- 排盘
- 规则命中
- 知识引用

---

# 六、知识库原则

## 6.1 区分数据类型

- rule（规则）
- example（案例）
- concept（概念）

---

## 6.2 检索必须混合

- 关键词
- 向量

---

## 6.3 chunk 不要过大

建议：
- 一个规则点一块
- 一个案例一块

---

# 七、数据与存储原则

## 7.1 必须保存快照

- 盘面
- 规则命中
- AI 输出

---

## 7.2 向量必须带版本

```text
model_name
provider
dim
version
```

---

## 7.3 不要过早上复杂架构

MVP：
- PostgreSQL
- pgvector

---

# 八、系统架构原则

## 8.1 单体优先

第一阶段：
- 1 个 Java 服务
- 1 个 Python worker

---

## 8.2 清晰边界

Java：
- 业务
- 规则
- 排盘

Python：
- 文档处理
- embedding

---

## 8.3 模型统一出口

禁止：
- 到处直接调用模型 API

必须：
- 统一 client

---

# 九、开发节奏原则

## 第一阶段
- 排盘
- 卦例
- 规则

## 第二阶段
- 知识库

## 第三阶段
- AI 分析

---

# 十、总结

这套原则的核心思想：

- 程序负责“算”
- 规则负责“落地”
- 知识负责“依据”
- AI 负责“解释”
- 数据负责“沉淀”

最终目标不是“算得最准”，而是：

**可解释 + 可验证 + 可复盘 + 可进化**
